Requirements

Build a robot-radar system robotAppl0 that includes a basicrobot (equipped with a robot-sonar), a userconsole and a radarGui so that:

    SPRINT1

  1. REQ-SHOW-DATA: the data produced by the robot-sonar should be properly shown by the radarGui.
  2. REQ-MOVE: an human user can move the robot by using the userconsole.

    3. SPRINT2

  3. REQ-TURN: the basicrobot must be able to detect obstacles and turn left of 90 degrees when an obstacle is detected.

    4. SPRINT3

  4. REQ-STEP: the basicrobot must be able to execute a command step(T) that moves the robot for the given time T.

Avoiding redirection

If we add to the default P2P qak-infrastructure the MQTT-based interaction support, any event can be directly perceived by any QActor of the system. Therefore a robotAppl0 system can be set-up by simply declaring the basicrobot and the radarGui with the specification of the MQTT support: 
mqttBroker "192.168.43.229" : 1883 

Context ctxBasicRobot	ip [ host= "localhost" port= 8018 ] +mqtt 
Context ctxRadarGui	ip [ host= "localhost" port= 8038 ] +mqtt
Now a console can be built as an 'alien' component that connects to the mqttBroker and then publish message in the qak standard representation on the topic unibo/qak/events. For example:

A prototype of a console

Let us introduce a prototype of a console in Python: 

import time
import paho.mqtt.client as paho
brokerAddr	= "localhost"
topic		= "unibo/qak/events"   #WARNING: infrastructural detail!!!
eventTemplate	= "msg(userCmd,event,python,none,userCmd(CMDVAL),1)" #qak msg-rep

def emit( cmd ) :
    message = eventTemplate.replace("CMDVAL", cmd)
    msg = message + "\n"
    client.publish(topic, msg)

def console() :  
    cmd =  str( input() )
    while( len(cmd)==1 and cmd != "q"  ) :
        emit( cmd )
        cmd =  str(input())
#################################################   
client= paho.Client("sender")      
client.connect(brokerAddr)             
console()

WARNING

The console code 'knows' the name of the topic used to emit/perceive qak events. This detail is allowable here, since we are just introducing a console prototype during system development. However:
The final console should consist in a user-GUI that completely hides any detail related to the interaction with the application.
This code can be executed by using the jupyter file Appl0ConsoleMqtt.ipynb

Towards systems based on the Subsumption architecture

Now, the actor robotappl0 of the robotAppl0 application is introduced just to 'extend' the behavior of the basicrobot. In other words, we could structure the behavior of the system by following the principles of Subsumption architecture.

SPRINT3

Requirements

REQ-STEP: the basicrobot must be able to execute a command step(T) that moves the robot for the given time T.
REQ-STEPABORT: if an obstacle is detected while performing a step(T), the robot must return to its initial position. The same must happen also when the user sends to the robot a command to stop.

Problem analysis

Now, the application layer does introduce its own way to move the the robot. As a consequence, the possibility to move the basicrobot by emitting usercmd events should be excluded. To achieve this goal:
  1. the application layer could be able to inhibit the behavior of the basicrobot (as suggested by the subsumption model) or to intercept usercmd events;
  2. the application layer could provide an user-console able to impose proper constraints to the interaction.
In the following, we will adopt the second strategy, by supposing that the console provided by the application interacts with robotappl0 in a direct way, using a Dispatch cmd : cmd(MOVE).

Towards a smart robot

Let us return to the idea of building a smart robot introduced in Lab3_2020 | A smartrobot.

Our smartrobot :
  1. Is an interpreter of commands of form cmd:cmd(MOVE) sent to it as a dispatch.
  2. Accepts also commands of form step:step(TIME) sent to it either as a dispatch or as a request.
    In the case of a request, it sends a reply of form stepdone:stepdone(ok) in case of success or of form stepfail:stepfail(DURATION, CAUSE) in case of failure.
    A step command can fail if an obstacle does not allow the move or if a command stop:stop(CAUSE) is sent to the robot as a dispatch.
  3. Updates a resource that keeps track of the robot position, when a step command is executed with success. Such a resource is implemented as a CoAP resource with url=coap://localhost:5683/robot/pos.
  4. Is able to start the CoAP resource in automatic way, together with a resourceObserver. However, another way to look at the resource evolution can be done also via 'alien' components, including Copper.
  5. Can receive commands from a 'conventional' console (for example a console written in python) or from a CoAP client that sends commands to the resource PUT verb.
To see some code: smartrobot.qak.

Incremental building

Coroutines, channels, actors

Do not communicate by sharing memory; instead, share memory by communicating
ActorBasic is a kotlin class that embeds an actor. It works in a message-driven way. It could be part of a context of type QakContext.

An actor is a coroutine builder a dual of produce coroutine builder. It is made up of a coroutine and a channel to communicate with other coroutines.

Coroutines are suspended functions, i.e. computations that can be suspended without blocking a thread.
send and receive are suspending functions used for providing and consuming (FIFO) objects from a Channel<T>

See LabKotlin | Introduction to KOTLIN: Actors

QActor

 QActor is an abstraction implements by the kotlin class ActorBasicFsm that extends ActorBasic It works as a FSM, i.e. in a message-based way.
It is part of a context of type QakContext and interacts with other QActor by means of dispatch, request or events.

See LabQakIntro2020.html

The robotSupport and the sonar pipe
  • robotAdapterQaStream: a QActor that provides an 'adapter-utility' to send commands the robot actuators and works as an observable sonar data source.

  • sonardh: an ActorBasic that eliminates 'wrong' sonar data
    See sonardsh.kt

  • filter: : an ActorBasic that detects obstacles and emits (using the robotAdapterQaStream) the obstacle event. It is the end of a built-in pipe that can be extended by the application designer.

  • sonarforradar: : an ActorBasic defined by the application designer that emits polar events.

The basicrobot layer

See LabNanoRobot | The basicrobot 		The basicrobot:
  • accepts commands for the actuators (motors) via MQTT or via the TCP-port 8018
  • handles sensor data (sonar) by using a reactive pipe (made of components of ActorBasic type) and emits events related to the perceived distance.
  • can be used by means of a console working as a TCP-client or as a MQTT-publisher

The smartrobot layer

The model: smartrobot.qak 		The smartrobot
  • works as a new layer built upon the basicrobot that adds the new command step.
  • can be used by means of a console working as a TCP-client (port 8020) or as a MQTT-publisher
